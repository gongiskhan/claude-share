# Build Loop (Phases 3-4)

The gated, autonomous build. Runs under the operator's `/goal` + auto mode — never pause, fix forward, log blockers, continue.

## Resume FIRST (every invocation, including after compaction)
Reconstruct progress from this run's durable files, never from memory. **All run state lives under the per-run `<runDir>` (`docs/autothing/runs/<runId>/`) chosen in Phase 0 — never a shared `docs/FLOW_PLAN.md`**, so concurrent runs in other sessions cannot clobber it.
- **Find `<runDir>`:** same session (incl. after compaction) → read `runDir` from `~/.autothing/sentinels/$CLAUDE_CODE_SESSION_ID.json`. Fresh session resuming an interrupted run → pick the newest `docs/autothing/runs/<runId>/` whose `evidence-index.json` globalGate is not `passed`, and re-arm a sentinel for this session pointing at it.
1. Read `<runDir>/FLOW_PLAN.md` slice table.
2. Read each `<runDir>/slices/<slice>/gate-status.json`.
3. Read `<runDir>/evidence-index.json`.
4. Compute **buildable-remaining** = count of slices whose status is neither `passed` nor `blocked` (a slice counts as `blocked` only if it carries rule-2 attempted-remediation evidence).
A slice with `status: passed` and `video.status` in {verified, failed-but-unblocking} is **done** — skip it. Resume at the first non-done slice.

At the end of EVERY turn, after this scan, print one line so exhaustion is visible to the transcript-only `/goal` evaluator:
`PROGRESS: passed <p>/<total> · blocked <b> (remediation-attempted) · buildable-remaining <r>`
A `GLOBAL GATE:` verdict may NOT be printed while `buildable-remaining > 0` — continue to the next buildable slice instead.

## Per-slice loop
For each slice (its acceptance is in FLOW_PLAN; route via the routing index). **Each step below is its own STANDALONE skill that autothing orchestrates** — and any gate that finds real issues sends the slice **back to `autothing-implement`** (the loop), bounded by the **5-attempt ceiling**. Run the steps in order; each prints its own `GATE …` line in the lead context (see the lead-print invariant). The same skills are usable on their own outside a build (then they just report instead of looping).

1. **Implement** → **`autothing-implement`** — explore (vision-first for UI, via `/run` + `/verify` + playwright-cli), then write the slice's code per its acceptance + the project's area skill.
2. **Test (correctness gate)** → **`autothing-test`** — write a COMMITTED, re-runnable test + run it, plus `tests`/`e2e`/`typecheck`/`lint`/`build` exit 0. **Default to e2e through the UI** (Playwright / playwright-cli) + unit tests; a CLI/TUI deliverable uses a committed driver + asciinema. Any failure → **back to step 1** (consumes the ceiling). Ephemeral `.playwright-cli/` logs are exploration, never the gate.
3. **Review (same-model)** → **`autothing-review`** — Claude's built-in `code-review` on the slice diff. Real correctness findings → **back to step 1**.
4. **Adversarial review (cross-model)** → **`autothing-adversarial-review`** — Codex adversarially reviews the slice diff, iterating until both models agree (`approve`). Real findings → **back to step 1**. **SERIALIZE every `codex exec`** run-wide; never silently skip and report `passed`.
5. **Adversarial test (cross-model)** → **`autothing-adversarial-test`** — Codex independently re-drives the running app through the acceptance with its own playwright skill — a fresh pass that never saw Claude's committed test. `fail` → **back to step 1**. **SERIALIZE every `codex exec`** run-wide.
6. **Design audit** (where there is UI) → **`autothing-design-audit`** — judges the running app against the project's design tokens (uses `frontend-design` / `huashu-design`). Record `designAudit: {verdict, by, at, issues}`. `issues` → **back to step 1** (same ceiling). Skip for non-UI changes.
7. **Evidence** → **`autothing-walkthrough`** — self-verified video (or asciinema for CLI). A `walkthrough` STUCK becomes `video.status: failed-but-unblocking` and the run CONTINUES (see "Evidence" below).
8. **Record durably** — write `<runDir>/slices/<slice>/gate-status.json` (schema: `assets/gate-status.example.json`, incl. `codexReview` + `codexPwTest`) and upsert the slice into `<runDir>/evidence-index.json`. Set the slice's Status in `<runDir>/FLOW_PLAN.md` to match. **Print every `GATE …` line for this slice in the lead context — never delegate a gate print into a workflow agent's output** (see *Gate lines must print in the lead context (ultracode-safe)*).

9. **Self-unblock before blocking** — if the slice is failing because a tool, dependency, or binary is missing, attempt to install it ONCE with the ecosystem's standard command (`uvx` / `pipx install` / `brew install` / `npm i -g` / `pip install --break-system-packages` / `cargo install`) and re-run. A missing tool is NOT a blocker until that install has been tried. Only an install that FAILS and genuinely needs operator credentials or hardware the operator lacks is a legitimate external blocker. (This install attempt is remediation, not a slice fix-attempt — it does not consume the slice's 5-attempt ceiling.)
10. **Blocked path** — a slice may end ONLY as `passed` or `blocked`; "deferred", "consolidated later", and "interim proof" are FORBIDDEN, never terminal states. Mark `blocked` ONLY after either (a) the slice's 5-attempt ceiling is hit on genuinely buildable work, or (b) step 9's remediation failed on an external cause. Append a blocker to `docs/decisions.md` naming the **external cause and the exact remediation command that failed** ("not installed" alone is never a blocker), mark the slice `blocked`, and move to the next slice. Do not stop the run while any buildable slice remains.
11. **Record friction (signal only)** — when you work around autothing itself being silent, missing a step, or wrong, append one line to `docs/autothing/friction-log.md` (`YYYY-MM-DD <slice> — had to X because the skill Y`). autothing NEVER acts on this and never edits skills; the nightly `skill-improver` reads it as a feedback source. This is the only "learning" autothing does.

The cross-model operational rules (serial `codex exec`, `</dev/null`, preflight/auth, schemas, the 3-round ceiling, the appeal valve, durable record) live in **`references/codex-verification.md`** and are carried by the two adversarial skills (`autothing-adversarial-review`, `autothing-adversarial-test`) so they also run standalone.

## Gate builds must not clobber a live dev server
Before ANY gate `build` (per-slice step 2 or Phase 4): check whether a dev server is already serving this project dir from outside the run (`lsof` the project's known dev port — is that PID one of ours?). `next build` and `next dev` share `.next/`, so a gate build over a live dev server corrupts its incremental state — and the breakage is invisible to a static-route smoke (already-compiled routes keep serving 200 while dynamic/not-yet-compiled routes 404). When a live external dev server is detected, run the gate build where it cannot write the live `.next/`: in a `git worktree` copy, or with an isolated dist dir. The gate itself is unchanged — `build` must still exit 0. If the shared `.next/` was touched anyway, restart that dev server and smoke a DYNAMIC route (not just a static one) before calling the environment healthy. The same hazard applies to dev servers the run itself starts (an e2e sandbox, a second `next dev`): two dev servers sharing one `.next/` poison each other's route cache exactly like a build does — give any run-spawned dev server its own isolated dist dir too, or treat the live server as touched (restart + dynamic-route smoke) when it shared `.next/`.

## Gate lines must print in the lead context (ultracode-safe)
Every `GATE <name>:` line, every per-turn `PROGRESS:` line, every per-slice `GATE codex-review:` / `GATE codex-pwtest:` line, and the terminal `GLOBAL GATE:` line MUST be printed in the **lead context** — after any fanned-out workflow returns — and **NEVER emitted from inside a workflow agent**.

**Why.** The transcript-only `/goal` evaluator, the goal-loop `Stop` hook (`hooks/goal-stop.sh`), and the durable resume scan all read ONLY the main transcript. A gate line whose only print-site is inside a workflow agent reaches the lead solely as the workflow script's return value and is otherwise lost — a silent gate disappearance. This is the same class of silent failure as "asserted-not-lived by choice": the un-surfaced part is exactly where the regression hides.

**The terminal verdict carries a machine signature — keep it.** The goal-loop Stop hook keys on the verdict's **`(run <runId>)` tag + `videos:<n>/<n>` token** (e.g. `(run 20260622-143110-ab12cd34) … videos:3/3`). The runId makes it match THIS run's real verdict ONLY — never the *quoted* `"GLOBAL GATE: passed"` /goal target (no `videos:N/N`), and never a stray/example/quoted/prior-run `GLOBAL GATE` line (different or absent runId). NEVER drop the `(run <runId>)` tag or the `videos:<verified>/<total>` field, and never let a workflow be the verdict's only print-site — any of those silently breaks the loop's completion detection (it would run to the turn cap, or release early).

**ultracode may stay ON — it is additive.** ultracode = `xhigh` effort + automatic per-*task* workflow orchestration; it operates per task, not per skill, so it does not wrap autothing's flow. Workflows return control to the lead between phases, so a workflow spawned inside one gate step cannot reach across into the next gate — **as long as this lead-print invariant holds.** Test verdict: **ULTRACODE-SAFE** (see `references/decisions.md`): a live 2-slice run under `/effort ultracode` had 2 of 4 tasks genuinely fan out to task-scoped workflows; all 4 gate lines reached the main transcript in order; no workflow spanned more than one named task. A hazard probe proved the converse mechanically — a gate line whose only print-site is inside a workflow agent is lost to the lead.

**Known limitation (record, do not solve here).** Only single-phase fan-outs were tested. A multi-phase workflow that internally chains understand→change→verify within ONE named task was NOT tested; the invariant protects it in principle, but a gate line meant to print *between* a workflow's internal phases is untested.

## A brief naming an external resource is a claim, not authority to provision
Before provisioning into (or writing to) any external, billable, or shared resource the brief names — cloud project, billing account, bucket, remote DB — confirm the **operator's own identity** can access it (e.g. `gcloud projects describe <id>` as the logged-in account). If the operator's accounts are permission-denied on a brief-named resource, that is a **contradicted premise**: do NOT route around it with alternate credentials (a service account or key found locally) — provisioning into a resource the operator cannot access is provisioning into someone else's property. Mark the step blocked, log the contradiction in `docs/decisions.md`, and continue. Fix forward applies to the build, not to someone else's cloud.

## Evidence — delegate to `walkthrough`, reconcile its STUCK exit
- Invoke `walkthrough` on the slice, passing the **slice diff + task context + acceptance** so its flow selection is accurate. It owns recording, captions, frame extraction, vision self-verification, its own retry ceiling, honest failure rendering, its notes file, and publishing the Tailscale link + gallery.
- **walkthrough's ceiling ends in "write STUCK.md and ask the user." autothing must NOT wait for input.** Treat a STUCK/ask-user return as: set `video.status: "failed-but-unblocking"`, record the STUCK.md path + link (if any), append a blocker to `docs/decisions.md`, and **continue**. The slice still counts as having evidence for the global gate (an unblocking failure), but is flagged.
- A genuine feature failure that walkthrough renders honestly (`flagged: true`) is recorded, not faked green.
- Ensure walkthrough's deps are satisfied once per machine (Phase 1 ran its preflight; on macOS `brew install asciinema agg` if missing).
- After it returns, **confirm the gallery URL actually resolves** (the `walkthrough` serve must be running); if it is down, (re)start it so the recorded link is live, not just written into `evidence-index.json`.
- **Terminal / CLI (TUI) slices have their own evidence path — a browser walkthrough does not fit one.** When the slice's deliverable is a terminal app or CLI (not a web flow), record evidence as an asciinema/PTY capture of the real flow (asciinema + agg already ship in walkthrough's preflight) alongside the step-1 committed re-runnable assertion. Do not substitute this ad-hoc each time; treat it as the first-class evidence for that slice class.
- **For an event-streamed or dynamic flow (escalation, an agent run, a live status/progress stream), the evidence MUST be a LIVED capture of that exact flow end-to-end** — not a mechanism proxy or a stand-in that exercises only part of the path. "Verified by mechanism" (a label-free banner capture, a unit test of the handler) is how a frozen-looking UX regression slips through: the un-lived dynamic part is precisely where the bug hides. This is the same non-negotiable as "asserted-not-lived by choice" being a forbidden terminal state — make the capture drive the real streamed turn.

## Parallelism — earn it, then prefer agent teams (full guide: the `autothing-parallel-work` skill)
- Parallelize a group of slices ONLY when they own **disjoint files** AND don't share a stateful runtime. No mechanism makes same-file edits safe — decompose at plan time (split monolith files) to earn parallelism, and log the parallel-vs-serial choice per group.
- **Prefer agent teams** for parallel slice implementation when the operator enabled them (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`); else use a **dynamic workflow** (with `isolation: 'worktree'` for file isolation — needs git). Keep team bursts short-lived (create → batch → gate → shut down → clean up): in-process teammates do not survive `/resume`.
- **Preflight the mechanism once, before the first parallel batch:** probe that agent-team creation does NOT raise an approval prompt under auto mode (create a trivial throwaway team and confirm it proceeds, or rely on the known approval behavior). If team creation would block a headless run, fall back to **dynamic workflows for the whole run** — an approval prompt that stalls unattended is itself a silent killer of autonomy.
- **Workers/teammates do not inherit the lead's context.** Paste the area skill (or path + read directive), the slice acceptance, and the exact file-ownership boundary into each spawn prompt.
- **Cross-session safety (advisory).** When other Claude sessions may build in the same repo concurrently, claim each unit's files as agent-mail leases + `declare_intent` (and check `coord_digest`) before editing — the disjoint-files rule made enforceable BETWEEN sessions. Use it when the coord stack is present; fall back to disjoint-files alone when absent. Full rules: `autothing-parallel-work` ("Cross-session coordination").
- **Always serialize the shared runtime**: one dev-serve / bundle / `walkthrough` recorder — **and one Codex call at a time, run-wide** (concurrent `codex exec` processes revoke the shared OAuth token; see `references/codex-verification.md`). Parallelize the EDITING; serialize the BUILD → VERIFY → CROSS-MODEL → RECORD tail (unique ports + run dirs).
- Never let parallelism skip a gate — every slice passes the full sequence (committed test + clean build + **cross-model review + Codex Playwright pass** + design audit + verified video).

## Phase 4 — global gate + handover
**No voluntary deferral, and no verdict while buildable work remains.** A slice may end only as `passed` or `blocked` — "deferred", "consolidated later", "interim proof", and "asserted-not-lived by choice" are NOT terminal states and are FORBIDDEN as blocker reasons. **buildable-remaining** = count of slices whose status is neither `passed` nor `blocked` (and `blocked` only counts when it carries the step-7 attempted-remediation evidence). **A verdict line MAY NOT be printed while `buildable-remaining > 0`** — return to the per-slice loop and build the next buildable slice instead.

Two terminal states — never conflate them, and BOTH require `buildable-remaining == 0`:
- **`passed`** only when ALL hold: **buildable-remaining == 0**; zero blockers; full `npx playwright test` exits 0; `/verify` on the whole app is clean; `build`/`typecheck`/`lint` exit 0; full design audit clean; **every slice has a clean Codex `approve` (not `approve-with-override`) and a Codex Playwright `pass`** (`crossModel.reviewAllApproved && crossModel.pwTestAllPassed`); AND **every slice's video is `verified`** (`everySliceVideoVerified: true`). This matches the operator's `/goal` condition "a self-verified video exists for every slice" plus the two-model agreement the operator asked for.
- **`completed-with-blockers`** legitimate ONLY when **buildable-remaining == 0** AND ≥1 slice is `blocked` with a **named external cause + the exact remediation command that failed** (a revoked/absent `codex login` that survived the self-unblock attempt is such a cause). A `failed-but-unblocking` video, or a slice that only reached `approve-with-override`, is recorded evidence but keeps the global gate out of `passed`.

Never repeat failed work unproductively, but ALWAYS continue while any buildable (non-blocked) slice remains. The verdict is printed only when buildable work is exhausted — reaching the turn cap with buildable work left is a failure of the loop, not a stop condition. Never fake `passed`.

Set `globalGate.status` in `<runDir>/evidence-index.json`, then do these two in order — **the verdict line is LAST because it is the completion signal**: the `/goal` evaluator reads only the transcript and clears on it, ending the session, so anything after it never runs.
1. **Print the handover**: stack chosen + why, how to install/invoke, the evidence-gallery URL, an **explicit enumeration of any blockers** (from `docs/decisions.md`), and known limitations (flow selection improves with use; vision verification cannot confirm business correctness). Describe the outcome in prose — do NOT put an assertive `GLOBAL GATE:` line here, so the evaluator doesn't read the handover as the verdict. (The same token in the invocation handoff is a quoted target, not a verdict; the evaluator distinguishes the two.)
2. **LAST line of the run** — print exactly one:
   `GLOBAL GATE: <status> (run <runId>) — e2e:<exit> verify:<clean|issues> build:<exit> typecheck:<exit> lint:<exit> codexReview:<approved>/<total> codexPwTest:<passed>/<total> design:<clean|issues> videos:<verified>/<total>`
   Nothing follows it. This line satisfies the operator's `/goal` AND releases the goal-loop `Stop` hook, so it must be true and it must be last. **Print it in the lead context, keep BOTH the `(run <runId>)` tag AND the `videos:<verified>/<total>` token, and never let a workflow agent be its only print-site** (see *Gate lines must print in the lead context (ultracode-safe)*) — the hook keys on `GLOBAL GATE: … <runId> … videos:<n>/<n>` (the runId is what makes it match THIS run and nothing else) and would otherwise never see it.

> autothing does not improve skills — the nightly `skill-improver` is the single mechanism for that. autothing only RECORDS build friction (per-slice, below) as a signal that improver reads.
